📊 EMR 가독성 가이드
환자 히스토리, 어떻게 읽어야 빠를까?
EMR은 모든 데이터가 있지만 읽기 어렵다는 게 문제다.
한 환자의 5년 치료 기록을 30초 안에 파악하려면 — 시각화 구조가 달라야 한다.
V
V-모델 (V-Model)
증상 → 검사 → 진단 → 처치 → 결과, 인과 사슬로
핵심 아이디어: 날짜순으로 쭉 나열하는 게 아니라, 같은 임상 문제(치아·부위)별로 묶어 그 안에서 주소(CC) → 현병력(PI) → 검사·진단 → 처치 → 결과·예정을 화살표로 이어 붙인다. "이 치료를 왜 했나"가 한 줄로 읽힌다.

🗣 주소 CC ➞ 🔬 현병력 PI ➞ 🔍 검사·진단 ➞ 🦷 처치 ➞ ⏭ 결과·예정
인과 한눈에 "왜 했나" 추적 일반 타임라인보다 추론 빠름(검증된 모델)

치과에 적용하면:
❌ 기존 방식
레진 / 신경치료 / 스케일링 / 크라운… 여러 치아 사건이 날짜순으로 뒤섞여 나열. "#36을 왜 신경치료까지 갔지?"를 알려면 흩어진 줄을 머릿속에서 다시 짜맞춰야 함.
✅ V-모델 방식
#36 시림 호소(CC) → 검사(PI) → 비가역적 치수염 진단 → 근관치료 → 크라운이 한 사슬로 연결. 원인부터 결과까지 한 줄로 읽힘.

실무 포인트: 치아(또는 임상 문제) 단위로 묶고, 각 사건에 단계 색을 부여(분홍=증상, 주황=검사, 노랑=진단, 파랑=처치, 보라=예정). → 실제 V-Model 시안 보기
↕
확장행 (Expandable Row)
요약 1줄 → 클릭하면 상세 펼침
핵심 아이디어: 기본은 최소 요약(날짜·치아·술식), 필요한 행만 펼쳐 메모·이미지·처방을 봄.

정보 과부하 차단 스캔 속도 향상

왜 효과적인가: 인간의 시선은 "전체 훑기 → 관심 항목 드릴다운" 순으로 움직인다. 모든 정보를 펼쳐두면 뇌가 필터링에 에너지를 낭비함.

설계 예시:
날짜치아술식금액
2025.03.12#36신경치료 2차 ▼35,000
📝 메모: 치근단 병소 확인, #36 MB2 발견. 파일 #25 사용.
💊 처방: 아목시실린 250mg×7일, 이부프로펜 400mg
📷 방사선: 수술 전후 PA 2장 첨부됨
2025.02.28#36레진 수복 ►80,000

이 차트에서: 각 방문 행 클릭 시 상세가 펼쳐지는 현재 구조가 바로 확장행 패턴.
≡
스윔레인 형식 (Swimlane)
진료 카테고리별 평행 트랙
핵심 아이디어: 수영장 레인처럼 카테고리(보존·보철·치주·교정·외과)를 수평 트랙으로 분리해 시간 흐름을 동시에 봄.

치료 간 인과관계 파악 미완료 치료 추적

왜 강력한가: #36 신경치료와 #36 크라운이 서로 다른 레인에서 같은 시간대에 진행됨을 한눈에 봄. 크라운이 왜 지연됐는지, 잇몸치료와 보철이 얼마나 겹쳤는지 즉시 파악.

레이아웃 예시:
보존
레진 #46 ──────── 신경 #36 ×3회
보철
───── 크라운 #36
치주
스케일링 ── 잇몸치료 #36
← 과거 ─────────────────────────────────── 최근 →

리콜 연결: 스윔레인에서 "잇몸치료 레인이 끊겼다" = 리콜 누락 신호. 레인 공백이 눈에 즉시 보임 → 리콜 타깃 자동 식별 가능.
⏳ 추가 예정: 히트맵 뷰 / 치아별 개인 타임라인 / 비용 흐름 시각화
